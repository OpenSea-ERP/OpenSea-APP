### Critério 1 — HTML Semântico (h1–h6, landmarks)

**Resultado: WARN**

#### Evidências positivas

- A maioria das páginas de entidade (fabricantes, categorias, volumes, inventário, pedidos de compra) utiliza corretamente `<h1>` para o título principal da página.
- O componente `PageHeader` (`src/components/stock/page-header.tsx`, linha 128) renderiza `<h1>` como elemento raiz de título.
- O layout do dashboard (`src/app/(dashboard)/layout.tsx`) define o elemento `<main id="main-content">` (linha 108), provendo o landmark `main`.
- A paginação (`_shared/components/pagination.tsx`, linha 82–86) usa `<nav aria-label="Paginação">`, atendendo ao critério de landmark para navegação repetida.
- A navegação lateral do modal `ItemEntryFormModal` usa `<nav>` corretamente (linha 346).

#### Problemas encontrados

**1.1 — Hierarquia de cabeçalhos quebrada em `templates/[id]/page.tsx`**
`src/app/(dashboard)/(modules)/stock/(entities)/templates/[id]/page.tsx`, linhas 99 e 155:

```tsx
<h2 className="text-2xl font-semibold mb-2">  {/* estado de erro/loading */}
<h1 className="text-xl font-bold">{template.name}</h1>
```

O `<h2>` no estado de carregamento/erro aparece sem um `<h1>` precedente na página. O `<h1>` está dimensionado visualmente como `text-xl`, enquanto o `<h2>` usa `text-2xl`, invertendo a hierarquia visual.

**1.2 — Uso semântico equivocado de `<h3>` em listas de propriedades**
`src/app/(dashboard)/(modules)/stock/(entities)/manufacturers/[id]/page.tsx`, linhas 154–290:
Múltiplos `<h3>` são usados para rotular seções de propriedades (`CONTATO`, `ENDEREÇO`, `PRODUTOS`), mas não há `<h2>` intermediário que estruture a página abaixo do `<h1>` do fabricante. A hierarquia salta de `<h1>` direto para `<h3>`.

**1.3 — Ausência de landmark `<main>` implícito nas páginas de detalhe standalone**
`src/app/(dashboard)/(modules)/stock/(entities)/products/[id]/page.tsx`, linha 92:

```tsx
<div className="min-h-screen from-purple-50 ...">
```

O conteúdo da página de detalhe do produto é um `<div>` sem semântica; o `<main>` é herdado do layout pai, porém `showHeader={true}` do `ProductViewer` renderiza novamente um `<h1>` interno sem que haja `<header>` explícito, tornando difícil para leitores de tela identificar a estrutura.

**1.4 — Seções de filtro não marcadas**
Em `_shared/components/stock-filter-bar.tsx` e `search-section.tsx`, os grupos de filtros são `<div>` simples. Recomenda-se o uso de `<fieldset>` e `<legend>` para agrupar os controles relacionados.

#### Correções recomendadas

- Uniformizar a hierarquia `h1 → h2 → h3` em todas as páginas de detalhe.
- Em `templates/[id]/page.tsx`, substituir o `<h2>` do estado de erro/loading por uma `<p>` ou marcá-lo como `<h1>`.
- Envolver grupos de filtros em `<fieldset>/<legend>` nos componentes `ProductsFilterBar`, `VariantsFilterBar` e `ItemsFilterBar`.

### Critério 2 — ARIA em Elementos Interativos

**Resultado: WARN**

#### Evidências positivas

- Componente `PageHeader` (`src/components/stock/page-header.tsx`, linha 81): botões ícone no modo mobile têm `aria-label={text || 'Acao'}`.
- Botão "Voltar" em `PageHeader` (linha 123): `aria-label="Voltar"`.
- Botão "Copiar ID do produto" em `ProductVariantsItemsModal` (linha 383): `aria-label="Copiar ID do produto"`.
- Botão "Limpar seleção" em `ItemsActionBar` (linha 158): `aria-label="Limpar seleção"`.
- Botão ícone "Rotacionar corredor" em `properties-panel.tsx` (linha 150): `aria-label="Rotacionar corredor"`.
- Botões de incremento/decremento em `step-dimensions.tsx`: `aria-label="Diminuir número de corredores"` / `aria-label="Aumentar número de corredores"`.
- Componente `BinSelector` usa `role="combobox"` e `aria-expanded` (linha 175).
- Botão de remoção de similar em `variant-form.tsx` (linha 1002): `aria-label="Remover item semelhante"`.
- Paginação (`pagination.tsx`): todos os botões têm `aria-label` e `aria-current="page"`.
- `EditorToolbar` usa `<TooltipProvider>` com `TooltipContent` em cada botão ícone.

#### Problemas encontrados

**2.1 — Botão "Fechar" sem `aria-label` em `RenameProductModal`**
`src/app/(dashboard)/(modules)/stock/(entities)/products/src/modals/rename-product-modal.tsx`, linha 60:

```tsx
<Button variant="ghost" size="sm" onClick={onClose}>
  <X className="w-4 h-4" />
</Button>
```

O botão X de fechar o modal não tem `aria-label`. Para um leitor de tela, esse botão será anunciado apenas como "botão" sem nenhum rótulo descritivo.

**2.2 — Botões de ação em `ItemsActionBar` usam `title` mas não `aria-label`**
`src/components/stock/... items-action-bar.tsx`, linhas 62–143:
Os botões de ação ("Vender", "Utilizar", "Devolver", "Transferir", "Desvio", "Reserva", "Etiqueta") têm `title="..."` mas nenhum `aria-label`. O atributo `title` não é lido de forma confiável por todos os leitores de tela (particularmente no modo touch). Esses botões mostram apenas texto na viewport desktop (`hidden sm:inline`), mas no mobile ficam apenas com ícone sem label acessível.

**2.3 — Botões de download e exclusão em `attachment-list.tsx` sem `aria-label`**
`src/components/stock/attachments/attachment-list.tsx`, linhas 86–101:

```tsx
<Button type="button" variant="ghost" size="icon" className="h-8 w-8">
  <MdDownload className="h-4 w-4" />
</Button>
/* ... */
<Button type="button" variant="ghost" size="icon" ...>
  <MdDelete className="h-4 w-4" />
</Button>
```

Ambos os botões (download e excluir anexo) não têm `aria-label`. Um leitor de tela não tem como distinguir a ação de cada um.

**2.4 — Botão de limpar busca em `SearchSection` sem `aria-label`**
`src/components/stock/search-section.tsx`, linha 58:

```tsx
<button onClick={handleClearSearch} className="absolute right-3 ...">
  <X className="w-5 h-5" />
</button>
```

O botão `<button>` nativo de limpar o campo de busca não tem `aria-label`. Será anunciado como "botão" vazio.

**2.5 — `AlertItem` clicável sem semântica de botão**
`src/app/(dashboard)/(modules)/stock/_shared/components/stock-alerts.tsx`, linhas 66–78:
Quando `onClick` é fornecido, o `<div>` se comporta como botão, mas não tem `role="button"`, `tabIndex={0}`, `onKeyDown` nem `aria-label`. Usuários de teclado não conseguem ativar o alerta.

**2.6 — `BinCell` clicável sem semântica de botão**
`src/app/(dashboard)/(modules)/stock/(entities)/locations/src/components/zone-map/bin-cell.tsx`, linhas 196–237:
O `<div>` da célula de nicho tem `onClick` mas não tem `role="button"`, `tabIndex={0}` nem `aria-label` descrevendo o nicho. O mapa de nichos é inacessível por teclado.

**2.7 — `Label` sem `htmlFor` associado em `step-code-pattern.tsx`**
Classificado também no Critério 6; o `<Label>` que serve de rótulo para o grupo de Radio (`Separador`, `Dígitos do Corredor`, etc.) não está associado a nenhum input. Isso é ambíguo para leitores de tela.

#### Correções recomendadas

- `RenameProductModal`: adicionar `aria-label="Fechar"` ao botão X.
- `ItemsActionBar`: substituir `title` por `aria-label` em todos os botões de ação.
- `attachment-list.tsx`: adicionar `aria-label="Baixar anexo"` e `aria-label="Excluir anexo"`.
- `SearchSection`: adicionar `aria-label="Limpar busca"` ao botão X.
- `AlertItem`: converter o `<div>` para `<button type="button" aria-label={alert.message}>` quando `onClick` for fornecido, ou adicionar `role="button"` + `tabIndex={0}` + `onKeyDown`.
- `BinCell`: adicionar `role="button"`, `tabIndex={0}`, `aria-label={bin.address}`, `aria-pressed={isSelected}` e handler `onKeyDown`.

### Critério 3 — Navegação por Teclado

**Resultado: WARN**

#### Evidências positivas

- Todos os componentes shadcn/ui utilizados (Dialog, AlertDialog, Select, Switch, Tabs, Popover, DropdownMenu, RadioGroup) têm suporte nativo a teclado via Radix UI.
- Componentes de formulário (Input, Textarea, Label, Select) são elementos HTML nativos ou wrappers sobre eles.
- O `BinSelector` usa `Popover` do Radix, que gerencia foco automaticamente.
- O `SearchInput` no `BinSelector` tem `autoFocus` ao abrir o popover.
- Modais de edição de produto (`RenameProductModal`, `ItemEntryFormModal`) têm `autoFocus` no primeiro campo.
- Botões de incremento/decremento em `step-dimensions.tsx` são `<Button>` normais (focusáveis por padrão).

#### Problemas encontrados

**3.1 — `BinCell` não é navegável por teclado**
Conforme identificado no Critério 2, o mapa de nichos (`zone-map/bin-cell.tsx`) usa `<div onClick>` sem suporte a Tab/Enter. O mapa inteiro de localizações é inacessível sem mouse.

**3.2 — `AlertItem` clicável não é ativável por teclado**
Conforme Critério 2.5.

**3.3 — Tabs de seção em `VariantForm` são `<button>` nativos sem indicador de foco visual explícito**
`src/components/stock/variants/variant-form.tsx`, linhas 290–307:

```tsx
<button
  type="button"
  onClick={() => setActiveSection(section)}
  className={`px-4 py-2 cursor-pointer text-sm font-medium transition-colors ${
    activeSection === section
      ? '  text-blue-600 dark:text-blue-400'
      : ' text-gray-600 ...
  `}
>
```

Embora sejam `<button>` nativos (focusáveis por padrão), o espaço antes das classes no estado ativo (`'  text-blue-600'`) é um artefato de formatação. Mais importante: não há classe `focus-visible:ring` explícita, dependendo apenas do outline nativo do navegador que pode ser suprimido pelo CSS global.

**3.4 — Botões de seção em `ItemEntryFormModal` sidebar são `<button>` sem indicador de foco**
`src/app/(dashboard)/(modules)/stock/(entities)/products/src/modals/item-entry-form-modal.tsx`, linhas 348–393: mesma situação do item 3.3.

**3.5 — Seleção de tipo de entrada em `EntrySection` não é acessível por teclado de forma semântica**
Linhas 507–538: os cartões de seleção de tipo são `<button type="button">`, o que é correto. No entanto, não há `aria-pressed` ou `role="radio"` indicando o estado selecionado. Um usuário de leitor de tela não sabe qual tipo está selecionado.

#### Correções recomendadas

- `BinCell`: implementar suporte completo a teclado (Tab, Enter, Space).
- `AlertItem`: converter para elemento interativo quando `onClick` estiver presente.
- `VariantForm` e `ItemEntryFormModal` — tabs de seção: adicionar `role="tab"` e o `tablist` correspondente, ou pelo menos `aria-pressed={activeSection === section}` se mantidos como botões.
- Cartões de tipo de entrada em `EntrySection`: adicionar `role="radio"` + `aria-checked={isSelected}` dentro de um `role="radiogroup"` com `aria-label="Tipo de Entrada"`.

### Critério 5 — Contraste de Cores ≥ 4,5:1 (WCAG AA)

**Resultado: WARN**

#### Evidências positivas

- O sistema de design usa tokens CSS (`--os-gray-*`, `--os-blue-*`, etc.) com valores bem definidos.
- A maioria dos textos usa `text-muted-foreground` (aproximadamente `gray-500`, ~4,6:1 sobre branco) ou `text-foreground` (escuro), atendendo ao critério AA.
- Modo escuro com `dark:text-white` sobre fundos escuros (`dark:bg-slate-800/60`) garante contraste adequado.
- A diretiva `@media (prefers-reduced-motion: reduce)` está implementada no `globals.css` (linha 6).

#### Problemas potenciais encontrados

**5.1 — Texto de indicador de tendência em `KpiCard` com verde sobre branco**
`src/app/(dashboard)/(modules)/stock/_shared/components/kpi-card.tsx`, linha 82:

```tsx
<span className={trend.isPositive ? 'text-green-600' : 'text-red-600'}>
```

`text-green-600` (#16a34a) sobre fundo branco tem contraste de aproximadamente 4,5:1 — no limite do critério AA. Em fundos levemente coloridos (como `bg-green-100`), pode cair abaixo do mínimo.

**5.2 — Texto em `OccupancyText` para nível `empty`**
`locations/.../bin-cell.tsx`, linha 40: `text-gray-400` no modo claro (`#9ca3af`) tem contraste de apenas ~2,8:1 sobre fundo branco — abaixo do requisito AA (4,5:1 para texto normal, 3:1 para texto grande). Para células compactas com fonte de 8px, o critério é ainda mais rigoroso.

**5.3 — Texto "fora de linha" e itens em `bin-cell` no modo `detailed`**
Linhas 165–179: `text-[8px]` e `text-[9px]` com `opacity-70` e `opacity-50` sobre fundos coloridos. Fontes abaixo de 14px com opacidade reduzida praticamente garantem falha de contraste.

**5.4 — Rótulo da página de fabricante com `text-white/60`**
`manufacturers/[id]/page.tsx`, linha 154: `text-white/60` (white com 60% de opacidade) resulta em contraste de ~2,5:1 sobre fundos escuros médios — insuficiente para texto normal.

**5.5 — Placeholder em `search-section.tsx` e `SearchInput`**
Placeholders com `text-gray-500 dark:text-gray-400` são padrão do shadcn/ui e estão isentos da regra de contraste 4,5:1 (WCAG considera placeholders como texto inativo). Sem problemas aqui.

#### Nota

Uma auditoria completa de contraste requer ferramentas de renderização visual (Lighthouse, axe DevTools). Os itens acima são baseados em análise estática dos tokens de cor.

#### Correções recomendadas

- `KpiCard`: substituir `text-green-600` por `text-green-700` (≥7:1 sobre branco).
- `BinCell`: usar tamanho de fonte mínimo de 12px ou aumentar o contraste dos textos em modo `compact`/`detailed`. Substituir `text-gray-400` por `text-gray-500` no mínimo.
- `manufacturers/[id]/page.tsx`: substituir `text-white/60` por `text-white/80` ou `text-slate-300`.

### Critério 9 — Sem `outline: none` Sem Substituto Visual de Foco

**Resultado: WARN**

#### Evidências

Foram encontradas duas ocorrências de `outline: none` no `globals.css`:

**9.1 — Células de edição do componente Spreadsheet (linha 975)**

```css
.Spreadsheet__cell input,
.Spreadsheet__data-editor input {
  outline: none !important;
}
```

Aplica-se ao componente de planilha usado no módulo de importação. As células de edição da planilha não têm indicador de foco visível alternativo. Quando um usuário navega pela tabela com teclado e entra no modo de edição de uma célula, não há feedback visual de foco.

**9.2 — Editor ProseMirror (linha 1654)**

```css
.ProseMirror {
  outline: none;
}
```

O editor de texto rico (TipTap/ProseMirror), utilizado no módulo de e-mail, remove o outline sem substituto visível. Usuários de teclado não sabem quando o editor está focado.

#### Contexto

O framework TailwindCSS 4 e o shadcn/ui aplicam `focus-visible:ring-2 focus-visible:ring-ring` em todos os componentes interativos (Button, Input, Select, Switch, Checkbox, etc.). As classes CSS globais do shadcn/ui, conforme verificado, não suprimem o `outline` dos elementos nativos. As supressões são pontuais e limitadas a contextos específicos de terceiros.

Os componentes de tab de seção em `VariantForm` e `ItemEntryFormModal` (identificados no Critério 3.3) também carecem de `focus-visible:ring`, confiando apenas no outline padrão do navegador que pode ser afetado.

#### Correções recomendadas

- `Spreadsheet` (`.Spreadsheet__data-editor input`): substituir `outline: none !important` por uma borda ou shadow customizada, por exemplo:
  ```css
  outline: none !important;
  box-shadow: inset 0 0 0 2px var(--spreadsheet-selected-border) !important;
  ```
- `ProseMirror`: adicionar um indicador de foco alternativo ao container pai:
  ```css
  .ProseMirror:focus-within {
    box-shadow: 0 0 0 2px hsl(var(--ring));
  }
  ```
- `VariantForm` e `ItemEntryFormModal` — botões de aba: adicionar `focus-visible:ring-2 focus-visible:ring-blue-500`.

## Notas Finais

O módulo stock demonstra boa atenção à acessibilidade nos componentes principais: formulários de entidade têm labels corretamente associados, modais baseados em shadcn/ui gerenciam foco automaticamente, o layout global implementa skip navigation, e a paginação tem suporte completo a ARIA. As falhas identificadas se concentram em:

1. Componentes visuais interativos que não foram projetados para teclado (mapa de nichos, células de alerta clicáveis).
2. Editor de layout (`properties-panel`) com painel de propriedades completamente sem associação label/input.
3. Botões ícone isolados que dependem apenas do atributo `title`, que não é universalmente suportado por tecnologias assistivas.
4. Supressão de outline em dois contextos de componentes de terceiros sem substituto visual.

A pontuação geral de **6,1/10** pode ser elevada para acima de 8,0 com a implementação das correções de prioridade crítica e alta listadas acima.
